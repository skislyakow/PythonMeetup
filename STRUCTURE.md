# PythonMeetup Bot — Структура проекта

## О проекте

Telegram-бот для Python-митапов. Позволяет гостям анонимно задавать вопросы спикерам, отслеживать программу мероприятия, а организаторам — управлять спикерами и расписанием.

---

## Технологии (рекомендация)

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| Язык | Python 3.12+ | Митап про Python |
| Бот | `python-telegram-bot` (v20.x, asyncio) | Самая популярная, active maintenance |
| БД | SQLite (через `peewee` или `sqlalchemy` + `aiosqlite`) | Для MVP не нужен сервер; легко поднять |
| Миграции | `peewee` или `alembic` | Контроль версий схемы |
| Config | `.env` + `python-dotenv` | Токены и настройки |
| Логи | `structlog` | Структурированные логи |
| Админ-панель (будущее) | FastAPI + htmx | Легко, без SPA |

## Роли и их возможности (MVP)

### 1. Гость (любой пользователь бота)
- `/start` — приветствие, краткое описание
- Посмотреть программу мероприятия
- Выбрать текущего докладчика
- Написать анонимный вопрос спикеру

### 2. Спикер
- Всё, что может гость
- Получить список вопросов к себе
- Ответить на вопрос (ответ уходит гостю)
- После выступления — вернуться в режим гостя

### 3. Организатор
- Всё, что может гость
- `/add_speaker <telegram_id>` — добавить спикера
- `/set_schedule` — задать программу (пошаговый диалог: бот спрашивает название доклада → время → спикера → повторяет)
- `/notify` — сделать рассылку
- `/broadcast` — массовое оповещение об изменениях

---

## Сценарии использования (ключевые)

```
1. Гость → /start → бот показывает приветствие и меню
2. Гость → "Задать вопрос" → выбирает докладчика → пишет вопрос → вопрос уходит спикеру
3. Гость → "Программа" → видит расписание докладов
4. Спикер → "Вопросы ко мне" → видит список вопросов → отвечает
5. Организатор → /add_speaker → добавляет user_id как спикера
6. Организатор → /set_schedule → обновляет программу → уведомление всем
```

## Разделение задач (3 разработчика)

### Разработчик A (я) — Ядро бота и инфраструктура
- Настройка проекта: структура, зависимости, `.env`, `config.py`
- `main.py` — точка входа, подключение handlers
- Модели БД (`user.py`, `event.py`, `question.py`)
- Миграции и инициализация БД
- Написание тестовой инфраструктуры
- **Результат:** заготовка, к которой подключаются хендлеры

### Разработчик B — Хендлеры гостя и спикера
- `handlers/guest.py` — `/start`, просмотр программы, выбор докладчика, отправка вопроса
- `handlers/speaker.py` — получение вопросов, ответы
- `services/questions.py` — логика маршрутизации вопроса к текущему спикеру
- `services/schedule.py` — получение программы из БД
- Инлайн-клавиатуры для навигации
- **Результат:** полноценный UX для гостя и спикера

### Разработчик C — Организатор, уведомления и администрирование
- `handlers/organizer.py` — добавление спикеров, пошаговое управление программой (диалог), рассылки
- `services/notifications.py` — оповещения при смене программы
- Валидация ввода (проверка дат, времени, форматов)
- Команда `/broadcast`
- Интеграционное тестирование всех сценариев
- **Результат:** панель управления организатора

## База данных (MVP)

**users**
| Поле | Тип | Описание |
|------|-----|----------|
| id | PK | Telegram user_id |
| role | str | guest / speaker / organizer |
| full_name | str | Имя (из Telegram) |

**events**
| Поле | Тип | Описание |
|------|-----|----------|
| id | PK | auto |
| speaker_id | FK → users | Кто выступает |
| title | str | Название доклада |
| start_time | datetime | Начало |
| end_time | datetime | Конец |

**questions**
| Поле | Тип | Описание |
|------|-----|----------|
| id | PK | auto |
| from_user_id | FK → users | Кто спросил |
| to_speaker_id | FK → users | Кому вопрос |
| text | str | Текст вопроса |
| answer | str | Ответ (null, пока не отвечен) |
| created_at | datetime | Когда задан |

## Риски и митигация

| Риск | Решение |
|------|---------|
| Бот недружелюбный | Приветственный текст + человечный тон |
| Слишком много уведомлений | Только важные события; подписка по желанию |
| Бот непонятный | `/help`, подсказки в каждом меню |
| Баги в маршрутизации вопросов | Привязать вопрос к `event_id`, проверять `now BETWEEN start AND end` |

## Как работать параллельно (важно!)

Разработчики **B и C НЕ ждут** A. План такой:

### Шаг 1 (делает A — 1 день)
- Создаёт репозиторий, структуру папок, `requirements.txt`, `.env.example`
- Скелет `main.py` (пустой dispatcher)
- **Модели БД** (`user.py`, `event.py`, `question.py`) — без них B и C не напишут код
- `config.py` — загрузка .env
- Пустые файлы-заглушки для хендлеров (чтобы B и C мержили без конфликтов)

→ **PR #1: «Скелет проекта + модели»** — B и C сразу мержат и пишут свой код.

### Шаг 2 (B и C — параллельно, A — интеграция)
- **B** пишет `handlers/guest.py`, `handlers/speaker.py`, `services/questions.py`, `services/schedule.py`, клавиатуры
- **C** пишет `handlers/organizer.py`, `services/notifications.py`
- **A** доделывает `main.py` (регистрация хендлеров)

### Шаг 3
- Code review, мерж, интеграционное тестирование
- Деплой на сервер / Railway / Fly.io
